The Big Picture

  • like any Julia package, the source files are under /src
  • meta data files such as tag -> DSID mapping are under /config
  • the main looper is called main_looper() and all the real actions are in there
  • the atomic unit of runnable task is called AnalysisTask

A typical workflow

Say we're trying to produce BDT score histogram for process ZZ:

  1. call prep_tasks("ZZ") (this returns a vector of AnalysisTasks)
  2. run main_looper over each of the tasks, you can use threading, or distributed parallelism.
  3. each main_looper would would return a dictionary of histograms
  4. merge all the result (reduce(mergewith(+), results_from_3))
  5. profit

WVZAnalysis.prep_tasks — Function
prep_tasks(tag; shape_variation="NOMINAL", scouting=false, kw...)

For construction a collection of AnalysisTasks given tag and job options, check out [@AnalysisTask].

All posible tags can be found in config/file_list.json, there's also a convinient variable WVZAnalysis.ALL_TAGS that keeps track of all processes in an exclusive mannar:

Example

julia> WVZAnalysis.ALL_TAGS
("Signal", "ZZ", "Zjets", "Zgamma", "ttbar", "WZ", "tZ", "ttZ", "tWZ", "VBS", "VH", "Others")
julia> all_nominal_tasks = mapreduce(prep_tasks, vcat, WVZAnalysis.ALL_TAGS);

julia> length(all_nominal_tasks)
768

WVZAnalysis.main_looper — Method
main_looper(task::AnalysisTask)

The main entry point for running the main looper for a given task, it's done in steps:

  1. destruct all the options from the task (see AnalysisTask):
    (; path, sumWeight, arrow_making, BDT_hist, isdata, 
     shape_variation, controlregion, sfsys) = task
  1. determine what output dict to prepare.
  2. call the lower level main_looper (which has all arguments explicitly laied out)

This function also serves a function barrier for performance reason, because we have so many different behaviors in the same loop function.
